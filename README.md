# Vault Badge Styles

Vault Badge Styles - плагин для Obsidian, который рисует заметки, папки, внутренние ссылки и внешние ссылки как аккуратные плашки с иконками и цветами.

Главная идея: визуальные правила живут в настройках плагина, а не во frontmatter заметок. Заметки остаются обычными Markdown-файлами, а внешний вид настраивается отдельно.

## Что умеет

- Настраивать стиль папок по пути.
- Указывать несколько путей, URL-префиксов или пар свойств внутри одного правила, чтобы не дублировать одинаковые стили.
- Каскадно применять стиль папки к вложенным файлам и папкам.
- Настраивать стиль конкретного файла по точному пути, файл может быть любого формата.
- Настраивать внешние ссылки по префиксу URL, например `tg://`, `tel:`, `https://vk.com/`.
- Настраивать значения свойств по паре `свойство + значение`, например `Status = done`.
- Выбирать свойства и значения из подсказок, собранных по заметкам vault.
- Использовать иконки из папок внутри vault: SVG, PNG, WebP, JPG и JPEG.
- Использовать встроенные иконки плагина для статусов, социальных ссылок и типовых технических категорий.
- Использовать emoji или любой короткий текст вместо файла иконки.
- Перекрашивать SVG-иконки: оставить оригинал, взять цвет текста или выбрать отдельный цвет.
- Задавать цвет текста и цвет фона плашки для каждого правила.
- Отключать применение цвета текста у конкретного правила, не удаляя выбранный цвет.
- Отключать применение цвета фона у конкретного правила, не удаляя выбранный цвет.
- Управлять общей прозрачностью фона плашек.
- Копировать существующие правила и редактировать копии независимо от оригинала.
- Сокращать пути внутренних ссылок в режиме просмотра.
- Сокращать вложенные теги в режиме просмотра.
- Применять стили в дереве файлов, заметках, свойствах, Bases, панелях с упоминаниями и заголовках вкладок.
- Автообновлять пути правил, если файл или папка переименованы внутри Obsidian.
- Искать правила в настройках.
- Проверять правила и иконки командами диагностики.
- Экспортировать и импортировать конфиг через `vault-badge-styles.config.json`.

## Установка через BRAT

1. Установи плагин **Obsidian42 - BRAT**.
2. Открой Command Palette.
3. Запусти команду `BRAT: Add a beta plugin for testing`.
4. Вставь адрес репозитория:

   ```text
   https://github.com/hubbleDJ/obsidian-vault-badge-styles
   ```

5. Выбери последнюю версию.
6. Включи **Vault Badge Styles** в списке Community Plugins.

BRAT устанавливает только основные файлы плагина:

```text
main.js
manifest.json
styles.css
```

Встроенные иконки для BRAT продублированы в `main.js` как generated fallback. Исходная папка `icons/` остается в репозитории и source archive релиза для ручной установки и разработки.

## Ручная установка

1. Скачай последний release.
2. Скопируй файлы и папку:

   - `main.js`
   - `manifest.json`
   - `styles.css`
   - `icons/`

3. Положи их в папку:

   ```text
   .obsidian/plugins/vault-badge-styles/
   ```

4. Перезапусти Obsidian.
5. Включи **Vault Badge Styles** в Community Plugins.

## Правила

В настройках есть четыре основных раздела. В каждом правиле можно задать один стиль и несколько matcher-ов: несколько путей, несколько URL-префиксов или несколько пар `свойство + значение`.

### Правила каталогов

Правило применяется к папке по точному пути. В одном правиле можно указать несколько папок с одинаковым стилем. Например:

```text
Development/Golang
Development/JavaScript
```

Если включено каскадное применение, стиль этой папки будет применяться к вложенным файлам и папкам. Более точное вложенное правило переопределяет родительское.

### Правила файлов

Правило применяется к конкретному файлу по точному пути. В одном правиле можно указать несколько файлов с одинаковым стилем. Расширение может быть любым:

```text
main.md
Project tasks.base
_Artifacts/Attachments/logo.png
```

Файловое правило сильнее каскадного правила папки.

### Правила внешних ссылок

Правило применяется, если ссылка начинается с указанного префикса. В одном правиле можно указать несколько префиксов с одинаковым стилем:

```text
tg://
tel:
https://vk.com/
```

Если подходит несколько правил, выигрывает самое длинное совпадение. Например, `https://vk.com/` сильнее, чем `https://`.

### Правила значений свойств

Правило применяется к отрендеренному значению свойства в Properties, Bases и похожих панелях.

Свойство обязательно: правило сработает только для точной пары `свойство + значение`. В одном правиле можно указать несколько пар с одинаковым стилем:

```text
Task status = done
Project status = done
```

Так `Статус = done` и `Фаза = done` могут иметь разные стили и не будут мешать друг другу.

## Иконки

Плагин поддерживает два источника иконок:

- **Иконка из папки** - плагин ищет файл иконки в заданных папках.
- **Emoji / текст** - значение вставляется как есть.

Папки поиска иконок задаются относительно корня vault. Пример:

```text
_Artifacts/Attachments/icons/me
_Artifacts/Attachments/icons
.obsidian/icons
```

После пользовательских папок плагин автоматически проверяет встроенную папку:

```text
.obsidian/plugins/vault-badge-styles/icons
```

Эта папка поставляется вместе с плагином при ручной установке. При установке через BRAT те же встроенные иконки доступны через generated fallback внутри `main.js`. Плагин читает иконки из папки рекурсивно, поэтому можно хранить встроенные иконки в подпапках. SVG-иконки можно указывать без расширения:

```text
Status/done
Status/in-progress
Status/create
Status/cancel
Development/golang
Development/python
Socials/telegram
Socials/youtube
location
```

Если в пользовательской папке есть иконка с таким же именем или относительным путем, она будет использована раньше встроенной.

Лучший вариант для интерфейсных иконок - SVG или PNG/WebP с прозрачным фоном. JPEG тоже поддерживается, но обычно хуже подходит для маленьких иконок.

Если в правиле указана иконка без расширения, например `golang`, плагин по старой совместимости ищет SVG:

```text
_Artifacts/Attachments/icons/me/golang.svg
_Artifacts/Attachments/icons/me/golang/golang.svg
_Artifacts/Attachments/icons/golang.svg
_Artifacts/Attachments/icons/golang/golang.svg
.obsidian/icons/golang.svg
.obsidian/icons/golang/golang.svg
```

Дополнительно плагин строит индекс всех файлов иконок во всех папках поиска. Поэтому вложенную иконку можно указать относительным путем без расширения, если это SVG:

```text
Development/golang
Status/done
Socials/telegram
```

Если нужен другой формат, укажи расширение явно:

```text
golang.png
person.webp
icons/project.jpg
```

## Цвета и плашки

Каждое правило может задавать:

- иконку;
- режим цвета SVG-иконки: оригинальный, как цвет текста или свой цвет;
- цвет текста;
- применять цвет текста или наследовать его от родительского правила/темы;
- цвет фона;
- применять цвет фона или наследовать его от родительского правила/темы.

Форма плашки, отступы и базовый вид переиспользуют стиль тегов текущей темы Obsidian. Цвета берутся из правила плагина.

Цвет можно выбрать через палитру или ввести вручную в соседнее поле HEX. Поддерживаются короткий `#ABC` и полный `#AABBCC` форматы.

Если у правила выключить **Красить текст**, выбранный цвет текста останется в настройках, но плагин не будет применять его к плашке. Это удобно для правил, где нужна только иконка или фон, а цвет текста должен наследоваться от более общего правила или темы.

Если у правила выключить **Заливать фон**, выбранный цвет фона тоже останется в настройках, но не будет блокировать каскад. Если выше есть подходящее родительское правило с включенным фоном, плашка унаследует фон от него; если родителя нет, плашка останется без заливки.

Перекраска иконки применяется только к SVG. В цветовых режимах SVG вставляется inline и красится через `currentColor`, поэтому прозрачный фон остается прозрачным. Простые SVG-анимации через `animate` и `animateTransform` сохраняются, но встроенные `script`, `style` и внешние ссылки внутри SVG удаляются перед вставкой. Сложные многоцветные SVG, PNG, WebP, JPG и JPEG лучше оставлять в оригинальном режиме.

Общая прозрачность фона плашек задается отдельным ползунком в настройках. Она применяется к внутренним ссылкам, внешним ссылкам, значениям свойств и дереву файлов.

## Сокращение путей

Плагин может сокращать внутренние ссылки в режиме просмотра.

Например:

```markdown
[[Development/Golang/Golang]]
```

В отображении будет видно только:

```text
Golang
```

Исходный Markdown при этом не меняется.

## Сокращение тегов

Плагин может сокращать вложенные теги в режиме просмотра.

Например:

```text
#todo/СрочноВажно
```

В отображении будет видно только:

```text
СрочноВажно
```

При клике по тегу поиск выполняется по исходному полному тегу.

## Где применяются стили

Плагин умеет применять стили в нескольких местах Obsidian:

- дерево файлов;
- заголовки вкладок;
- Reading View;
- свойства заметок;
- Bases;
- панели с упоминаниями и похожие панели;
- внешние ссылки.

Live Preview поддерживается частично. Полная поддержка Live Preview требует отдельной реализации через CodeMirror decorations.

## Команды

Плагин добавляет команды:

- `Vault Badge Styles: Rebuild icon/style index`
- `Vault Badge Styles: Refresh file explorer icons`
- `Vault Badge Styles: Refresh reading view links`
- `Vault Badge Styles: Preview matched paths`
- `Vault Badge Styles: Debug current file style`
- `Vault Badge Styles: Validate rules and icons`
- `Vault Badge Styles: Export config to vault root`
- `Vault Badge Styles: Import config from vault root`

## Экспорт и импорт

Команда экспорта создает файл в корне vault:

```text
vault-badge-styles.config.json
```

Команда импорта читает этот же файл и заменяет настройки плагина.

Экспортируются только настройки. Пользовательские файлы иконок не упаковываются в архив и должны лежать в vault отдельно. Встроенные иконки плагина лежат в `.obsidian/plugins/vault-badge-styles/icons` и поставляются вместе с плагином.

## Ограничения

- Плагин меняет только отрисовку интерфейса Obsidian. Markdown-файлы не переписываются.
- Автообновление путей при переименовании меняет только правила каталогов и файлов в настройках плагина.
- Полная поддержка Live Preview пока не реализована.
- Некоторые новые панели Obsidian могут потребовать отдельного адаптера, если у них отличается DOM-структура.
- Для обновлений через BRAT лучше выпускать новый tag/release, а не перезаливать старый.
